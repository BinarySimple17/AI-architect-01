# AI-ассистент для службы поддержки

---

## Проблема
- Служба поддержки ERP при получении обращения от пользователя не имеет единой базы решений запросов пользователей. Есть разрозненные How-To, отдельные базы знаний по направлениям.
- Текстовый поиск по ним малоэффективен из-за различий в формулировках.
- Базы знаний наполняются вручную и не всегда, часть решений остается только в самих обращениях.

## Цель задачи
Внедрить корпоративный ИИ-сервис, который :
1. Предлагает найденные подходящие или схожие решения.
2. Периодически дополняет свою базу знаний из закрытых обращений.

## Ключевые возможности (MVP)
| Функция | Описание | Бизнес-эффект |
|---------|----------|---------------|
| Проверка на дубли | При пополнении базы знаний чтобы не засорять одинаковыми или почти одинаковыми обращениями |  |
| Семантический поиск | Поиск по смыслу, а не по ключевым словам | Ускорение поиска готовых решений |
| Админ-панель | Мониторинг качества, ручная валидация | Контроль и дообучение системы |
| Локальное развёртывание | Все данные и модели внутри периметра компании | Соответствие политике безопасности |

## Ограничения
- Запрещена передача данных во внешние облака и публичные API.
- Все компоненты (модели, векторная БД, оркестрация) разворачиваются on-prem в Kubernetes.
- Интеграция с ERP — через штатный oData-интерфейс, без модификации ядра системы.
- Частота обновления данных — не реже 1 раза в сутки.


# Требования для реализации бизнес-процесса ИИ-ассистента службы поддержки

## 1. Контекст и бизнес-цель
В компании используется ERP-система SM для управлениями изменениями и поддержки системы ERP. Внутренняя СУБД ERP не поддерживает векторный поиск. Требуется внедрить локальный AI-сервис для:
- автоматического выявления семантических дубликатов при пополнении базы знаний ИИ-ассистента;
- семантического поиска по существующим материалам в базе знаний на основе описаний на естественном языке.
Все данные и вычисления остаются в on-prem периметре. Передача данных во внешние SaaS-сервисы запрещена.

## 2. Пользовательские сценарии
- **Сценарий 1: Семантический поиск.** Пользователь формулирует запрос. Система находит релевантные материалы, фильтрует материалы по направлениям поддержки, применяет правила ранжирования, генерирует текстовую сводку через LLM и возвращает структурированный ответ.
- **Сценарий 2: Ручное добавление записи в базу знаний** Администратор заполняет форму и запросом и предложенным решением. Система проверяет наличие дубликатов в базе знаний и предупреждает об этом. Администратор может подтвердить сохранение, несмотря на предупреждение.
- **Сценарий 3: Ручное удаление записи из базу знаний** Администратор заполняет форму и запросом. Система ищет обращения в базе знаний и показывает их. Дальше можно выбрать и удалить конкретное обращение.

## 3. Функциональные требования
- Настройка параметров запроса: `top_k`, `similarity_threshold`, `temperature`, `timeout_ms`.
- Кэширование результатов поиска с настраиваемым TTL.
- Административный интерфейс (Spring Thymeleaf) для мониторинга статуса ETL, управления кэшем, просмотра логов и ручной валидации пополнения базы знаний.
- Возврат метаданных выполнения: время отклика, статус ответа, `trace_id`.

## 4. Архитектурный стек
| Компонент | Технология | Назначение |
|-----------|------------|------------|
| Frontend/Admin | Java Spring Boot + Thymeleaf | Админ-панель, не участвует в основном потоке ERP |
| Backend / Gateway | Spring Cloud Gateway + Spring Boot | Маршрутизация, аутентификация, валидация, rate-limit |
| AI Service | Python (FastAPI/Flask) | Оркестратор RAG-пайплайна, логика ранжирования, управление запросами к моделям и БД |
| Vector DB | Qdrant | Хранение и поиск векторных представлений базы знаний (`requests`, `solution_chunks`) |
| SQL DB | PostgreSQL | Сырые справочники, метаданные, логи запросов, состояние ETL |
| Cache | Redis | Кэш результатов поиска |
| Model Serving | Qwen (embedding + LLM) | Внутренний сервис инференса (on-prem). Отдельные модели для эмбеддинга и LLM |
| ETL Worker | Python | Polling oData ERP, трансформация, вставка в PG |
| Embedding Service| Python | Эмбеддинг, чанкинг, проверка дубликатов, вставка в Qdrant |
| Secrets | HashiCorp Vault | Хранение и ротация учетных данных. Уже развёрнут в компании |
| Observability | Prometheus + Grafana | Метрики, дашборды, алерты |

Языковая политика: Java Spring для Gateway, админ-интерфейса и бизнес-логики маршрутизации; Python для AI Service, ETL и работы с моделями. Есть команды Java и Python - разработку можно разделить.

## 5. Пайплайн обработки запроса
1. **Прием запроса:** Gateway → AI Service (`POST /api/v1/ai/get_recommendation`). (Реализуем идемпотентность при запросах через X-Idempotency-Key и traceId, X-User-Id для идентификации пользователя gateway извлекает его из токена и пробрасывает)
2. **Кэш-проверка:** Вычисление хэша (`query + metadata_filters + config`). При попадании в Redis → возврат кэшированного результата.
3. **Эмбеддинг:** AI Service → Internal Model Serving → генерация вектора запроса.
4. **Векторный поиск:** AI Service → Qdrant (`search` с фильтрами по метаданным, `top_k`). Поиск по полным запросам и поиск по чанкам решений.
5. **Ранжирование:** Расчет финального скоринга (cosine similarity с коэфициентом для полного попадания запроса или чанка решения), удаление дубликатов (если чанк принадлежит `solution`, которое уже есть среди полных решений, то такой чанк пропускается), отсечение по `similarity_threshold`.
6. **Генерация ответа:** Prompt Template Factory → сборка контекста из запроса,  top_k полных решений запросов и top_k чанков решений  → вызов LLM → формирование текстовой рекомендации.
7. **Постобработка:** Асинхронное сохранение метаданных в PostgreSQL BackgroundTasks FastAPI, обновление Redis, возврат ответа Gateway → ERP/Frontend. Ответ пользователю должен уходить сразу после формирования результата, не дожидаясь записи в БД.
8. **Fallback:** При таймауте или ошибке LLM → возврат результатов векторного поиска с `status: degraded`. При пустом ответе Qdrant → `status: empty`.

## 6. Нефункциональные требования
- **Безопасность:** mTLS между внутренними контейнерами, RBAC на Gateway, шифрование данных в PG/Qdrant, интеграция с Vault для всех секретов.
- **Производительность:** Целевая латентность < 3 с для поиска, < 5 с для генерации LLM. Circuit breaker и retry с exponential backoff на вызовы Model Serving и Qdrant.
- **Надежность:** Идемпотентность запросов через `X-Request-ID`, graceful degradation при недоступности компонентов.
- **Версионирование API:** REST `/api/v1/...`, обратная совместимость контрактов.

## 7. Принятые архитектурные решения
- ERP-СУБД не поддерживает векторы → индексация и поиск вынесены в Qdrant.
- Модели обслуживаются внутри кластера K8s → исключена передача данных наружу.
- ETL работает по расписанию в режиме polling по oData с фильтром `LastModifiedDateTime`, `Status`. Для полученных новых обращений вычиляется вектор, ищется top-1 ближайший вектор в `requests`, сравнивается сходство с порогом и автоматически или записывается или отлоняется. Спорные случаи направляются на ручную валидацию.
- Redis используется с TTL: 15 мин для результатов поиска.
- Основной поток запросов инициируется ERP-системой. Thymeleaf-интерфейс предназначен только для администраторов и отладки.
